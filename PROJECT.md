## What this is

A private, 2-person web app for planning a Japan trip in **September 2026**: pin places on Google Maps, organize them into a day-by-day schedule, add notes. Built by the user + Claude Code, for the user and their partner only.

## Confirmed requirements

- **Users:** Exactly 2 people (user + partner). No public signup, ever.
- **Stack:** Next.js (App Router) on Vercel. Supabase (Postgres + Auth) as the only backend — no custom server.
- **Maps:** Google Maps JavaScript API + Places API only. **No Directions/Routes API** — stops are simple ordered pins, no travel-time routing between them (deliberate, to keep API cost/complexity down).
- **Sync:** Manual refresh is fine. No Supabase Realtime.
- **Scope:** Map pins organized into a day-by-day schedule, each stop has a note + category, optional start time/duration. Trip-prep to-do list (see "Decisions changed" — this was originally excluded, then explicitly requested). Still no budget tracking, no booking links/confirmation numbers.
- **Offline:** View-only offline support (see the saved itinerary without signal, e.g. on a train). Adding/editing while offline is explicitly out of scope (would need a write-queue/conflict-resolution system not worth the complexity here).
- **Mobile:** Must be installable via "Add to Home Screen" (PWA) on both iOS and Android.
- **Domain:** Registered at **Exabyte**. Live at subdomain `trip.benjamin0308.my` via a CNAME record pointing at Vercel.

## Decisions changed after initial planning

- **Auth switched from magic-link to email+password.** Originally planned as passwordless magic link (2 manually-created Supabase users, public signup disabled). User found checking email for every login annoying, so switched to `signInWithPassword`. The "2 users only, no signup" lock is unchanged — still enforced via Supabase Dashboard → Authentication → disable "Allow new users to sign up." Passwords were set manually per-user via the Supabase dashboard (not self-serve reset flow). The `/auth/callback` route (needed only for magic-link) was removed as dead code.
- **Session behavior:** Not per-browser-session — Supabase issues a refresh token in a cookie that `proxy.ts` (formerly `middleware.ts`, renamed per Next.js 16 convention) silently refreshes on every request. In practice each device stays signed in for weeks until explicit sign-out or cleared cookies.
- **GitHub repo already existed** at `https://github.com/simplicity0308/suiTracker.git` with an unrelated scaffolded app (tRPC + better-auth + Drizzle-style, 2 commits). Per user's explicit choice, this was **force-overwritten** with our project history — old commits are gone from the remote (branch renamed `master` → `main` to match GitHub default).
- **Checklists added back in, deliberately.** The original plan explicitly excluded checklists to keep scope tight. User later asked directly for a trip-prep to-do list ("buy insurance 1 month before, purchase tickets," etc.) — added as a flat, trip-level list (`todos` table), separate from day-by-day stops, not tied to a specific day/pin.

## Architecture

- **DB schema:** `trips`, `trip_members` (join table gating access), `days`, `stops`, `todos`, `profiles`. Full SQL in `supabase/migrations/0001_init.sql`. RLS restricts all access to rows where `auth.uid()` is in `trip_members` for that trip, via a `security definer` helper function `is_trip_member(trip_id)`. Later migrations, **all must be run manually in the Supabase SQL Editor** (not auto-applied) — confirmed all applied and working (`0002`–`0007`, verified via real end-to-end use: stop times, todos with day assignment/notes, added-by badges all working): `0002_add_stop_time.sql` (`stops.start_time`/`duration_minutes`), `0003_add_todos.sql` (new `todos` table, same RLS pattern via `is_trip_member`), `0004_add_todo_time.sql` (`todos.due_time`), `0005_add_profiles.sql` (new `profiles` table for "added by" labels + `todos.created_by`), `0006_add_todo_day.sql` (`todos.day_id`, nullable FK to `days`, `on delete set null` matching the `stops.day_id` pattern), `0007_add_todo_note.sql` (`todos.note`). No new migration needed for weather — it's derived entirely from existing `lat`/`lng`/`day_id`/`start_time` columns.
- **To-do day assignment:** a to-do shows in a day column if EITHER its `day_id` explicitly matches that day, OR (when `day_id` is null) its `due_date` matches the day's `day_date`. This exists because a prep to-do's due date is often *before* the trip even starts (e.g. "buy insurance a month ahead") and would never date-match any day — the "Show under day" select in `TodoList.tsx`'s add form lets the user force it under a specific day regardless of its due date. Matching logic lives in `AgendaBoard.tsx`'s `todosForDay()`.
- **Agenda settings collapsed by default:** `TripDatesForm` and the manual "Add a day" form were cluttering the Agenda page on every visit for what are occasional/setup actions — now behind a gear (⚙) button next to the trip name in `app/trip/agenda/page.tsx`, collapsed (`showSettings` state, default `false`) until clicked.
- **Delete confirmations:** stop and to-do removal now confirm first (native `confirm()`, same pattern `DayColumn.tsx`'s day-delete already used) — `DeleteStopButton.tsx` now takes a `name` prop for the confirm text, `TodoRow.tsx`'s remove button wraps `onDelete` in a confirm.
- **To-do notes:** `todos.note` (migration `0007_add_todo_note.sql`), same free-text pattern as `stops.note`. Added to the add form in `TodoList.tsx` and displayed in `TodoRow.tsx`, which was restructured from a single-line flex row to a stacked layout (checkbox + vertical content block, matching `StopCard.tsx`'s shape) so the note has room to wrap under the title/date instead of squeezing into one line.
- **Mobile audit fixes:** found via code review (no device/browser tool available here) after user reported "desktop looks good, haven't checked mobile." Two real bugs: (1) the "SuiTracker" wordmark added during theming pushed the header (wordmark + 3 nav tabs + sign out) past ~370–390px total width with no wrap/overflow handling — overflows on phones ≤375px wide. Fixed by hiding the wordmark below the `sm:` breakpoint (`hidden sm:inline-block` in `app/trip/layout.tsx`) — nav/sign-out (the functional parts) always stay visible. (2) `MapView.tsx` sized itself with `h-[calc(100vh-49px)]` — `100vh` is a known mobile Safari/Chrome pitfall (measured against the largest possible viewport, not the actual visible area once the address bar is showing), switched to `100dvh`. Also stacked `InstallPrompt.tsx`'s text/buttons vertically below `sm:`, and bumped `text-[10px]` badges to `text-xs` across `StopCard`/`TodoRow`/`DayColumn`/`MapView` for legibility. Confirmation dialogs, the `flex-wrap` forms, and `TimePicker` were checked and already work fine on mobile — no change needed there.
- **Fixed day-color collisions on the map.** `DAY_COLORS` (`lib/constants.ts`) was a fixed 8-color palette indexed via `dayIndex % 8` — with an 11-day trip, day 9 silently reused day 1's exact color (same for day 10/day 2, day 11/day 3). Combined with the pin's number label actually meaning "Nth stop within that day" (not the day number), a day-9 stop that happened to be its day's only stop showed as an identical red "1" pin to an actual day-1 stop — this is what the user was reporting as "why is day 9 showing as D1." Replaced the fixed palette with `getDayColor(index, total)` (`lib/constants.ts`), which spaces hues evenly around the color wheel based on the trip's actual day count, so colors never collide regardless of trip length. Also added the day's label to the map `InfoWindow` popup (`MapView.tsx`) so clicking any pin unambiguously confirms which day it belongs to, independent of color.
- **Configurable trip date range:** `lib/actions/days.ts`'s `setTripDatesAndGenerateDays(tripId, startDate, endDate)` — updates `trips.start_date`/`end_date` and auto-creates any missing `days` rows for that range (default label `Day N`, matched/skipped by `day_date` so existing custom-labeled days are never touched or duplicated), then re-sorts all days by date. UI: `components/agenda/TripDatesForm.tsx` on the Agenda page, replacing what used to be a read-only date line. The manual "Add a day" form still exists alongside it for one-off days.
- **"Added by" badges:** `stops.created_by` already existed and was already populated but never displayed; `todos.created_by` added alongside it (Phase 4 didn't have this column). Both resolve through a new `public.profiles` table (`id` → `auth.users.id`, free-text `label`) rather than querying `auth.users` directly — that schema isn't accessible via the normal client and this app deliberately never uses the service-role key. `profiles` must be seeded manually, same one-time-SQL pattern as `trip_members`: get each account's UUID from Authentication → Users, then `insert into profiles (id, label) values ('<uuid>', 'Ben'), ('<uuid>', 'Partner');` (label is free text — real names, email-derived, whatever's preferred). Badge shows on `StopCard`, the map's `InfoWindow`, and `TodoRow` — renders nothing if unseeded or `created_by` is null, no crash.
- **One trip, seeded manually** via `supabase/seed_trip.sql` (user filled in the real UUIDs after creating the 2 auth accounts). No self-serve "create a trip" UI by design — add one later only if a second trip actually happens.
- **Server Actions** (`lib/actions/*.ts`) handle all mutations. No REST API routes needed.
- **Supabase clients:** `lib/supabase/client.ts` (browser), `lib/supabase/server.ts` (server components/actions), both via `@supabase/ssr`.
- **Route structure:** `/login` (public), everything under `/trip/*` gated by `proxy.ts`. `/trip/agenda` = day-by-day list + day management. `/trip/map` = Google Map with search-and-pin.
- **Data fetching (Phase 4 change):** `/trip/agenda` and `/trip/map` are now client components fetching via `useTripData()` (`hooks/useTripData.ts`, TanStack Query) instead of Server Components — required for offline viewing (see below). Mutating components call `queryClient.invalidateQueries` after their server action resolves to refresh the shared cache; `revalidatePath` was removed from the actions since it's now a no-op (nothing server-rendered depends on this data anymore).
- **Offline support:** `QueryProvider` (`components/providers/QueryProvider.tsx`) persists the TanStack Query cache to `localStorage` (14-day max age), so the last-loaded itinerary renders even with no network. Service worker via `@serwist/turbopack` (this Next.js version defaults to Turbopack, which the older `@serwist/next` webpack plugin doesn't support) — route handler at `app/serwist/[path]/route.ts`, worker source at `app/sw.ts`. Google Maps domains (`googleapis.com`/`gstatic.com`) are explicitly excluded from all caching in `app/sw.ts` (ToS requirement) — the map itself won't work offline, only the itinerary data/list.
- **Critical PWA bug, fixed:** after the first real production deploy, the live site broke entirely in any browser tab that had visited it before that deploy (worked fine in incognito) — Next.js renames JS/CSS chunks on every build, and the old service worker was serving a stale cached HTML shell referencing chunk hashes that no longer existed on the server post-deploy. Fixed in `app/sw.ts` by adding an explicit `NetworkFirst` rule (`networkTimeoutSeconds: 4`) for document/navigation requests, placed ahead of `defaultCache` — page loads now always try the network first when online (so they always get the current build), only falling back to the cached shell when the network genuinely fails (true offline use, which still works). If this bug resurfaces, check that this rule is still first in `runtimeCaching` before touching anything else. Affected users must clear it manually one time (DevTools → Application → Service Workers → Unregister, then Clear site data) — this fix prevents it going forward but can't retroactively un-stick an already-broken tab.
- **Important:** `proxy.ts`'s route matcher excludes `/serwist` and `/manifest.webmanifest` from the auth gate — without this, an unauthenticated service-worker registration request gets redirected to `/login`, which browsers reject outright (`SecurityError: ... behind a redirect`). Hit this exact bug once; if the matcher regex is ever touched again, keep those exclusions.
- **PWA install:** `app/manifest.ts` + `appleWebApp`/`icons` metadata in `app/layout.tsx`. Icons are a hand-authored Mount Fuji SVG (`public/icons/`, sunrise-over-Fuji style, generated via `sharp`, no external image API). `components/layout/InstallPrompt.tsx` shows a real Install button on Android/Chrome (`beforeinstallprompt`) or manual "Add to Home Screen" instructions on iOS.
- **"Open in Google Maps"** — `lib/utils.ts`'s `googleMapsUrl()` builds a plain deep-link URL (no Maps JS API call, zero billing impact), used in `StopCard.tsx` and `MapView.tsx`'s InfoWindow.
- **Time picker:** native `<input type="time">` was replaced with a custom `components/agenda/TimePicker.tsx` (hour / 5-min-increment minute / AM-PM as three `<select>`s) — the native input's format/AM-PM display is locale- and browser-dependent and was inconsistent; plain `<select>`s render as native wheel pickers on iOS/Android and give full control over format.
- **"Next up" highlight:** `lib/utils.ts`'s `getNextUpcomingItem(days, stops, todos)` is the single source of truth for "what's coming up next across the whole trip" — combines each stop's parent-day `day_date` + `start_time`, and each to-do's own `due_date` + `due_time`, finds the soonest one `>= now`. **Ties (identical date+time) are broken by whichever row was `created_at` first**, per explicit user request. Returns `{ kind: "stop" | "todo", id }`. `AgendaBoard.tsx` and `TodoList.tsx` each call it (with the full `days`/`stops`/`todos` triple, since either page needs to know if the *other* page's item type actually won) and only render a highlight if the winning `kind` matches what that component renders — so at most one thing is ever highlighted "Next up" across the whole app. `DayColumn.tsx` separately shows a "Today" badge via `isToday(day.day_date)`. All of this no-ops gracefully if dates/times aren't set.
- **To-do list ("could be anything related to the trip," not just prep):** `todos` table, flat/trip-level (no `day_id`), optional `due_date` **and** `due_time` (same `TimePicker` component as stops, and participates in the shared "next up" computation above). Managed from its own tab (`/trip/todos`, `app/trip/todos/page.tsx`) between Map and Agenda in `NavTabs.tsx`. **Also interleaved into the Agenda's day columns**: `DayColumn.tsx`'s `buildEntries()` merges that day's stops with any to-dos whose `due_date` matches `day.day_date`, sorted by time first (todos and stops mixed chronologically) then falling back to each type's own `sort_order` for untimed items. To-dos are rendered via a shared `TodoRow.tsx` (extracted from `TodoList.tsx`) but are **not** part of the dnd-kit `SortableContext` — only stops are drag-reorderable; to-dos in the day view are still toggle/delete-able (calls the same `lib/actions/todos.ts` actions). To-dos with no `due_date`, or a date not matching any created day, only appear on the `/trip/todos` tab. `lib/actions/todos.ts` (`createTodo`/`toggleTodo`/`deleteTodo`), folded into `useTripData()` (`hooks/useTripData.ts`) so it shares the same query/cache/invalidation key as days/stops.
- **Day date label:** `DayColumn.tsx` now shows `formatDayDate(day.day_date)` (e.g. "— Sep 15") next to the day's label, when a date is set. `formatDayDate` in `lib/utils.ts` builds the `Date` from explicit y/m/d components rather than `new Date(dateStr)`, to avoid a timezone-shift-by-one-day bug on the raw ISO string.
- **Login page:** `components/login/TokyoBackdrop.tsx` — hand-authored inline SVG night-Tokyo skyline (fixed, full-bleed), reusing the exact gradient palette from the Mount Fuji PWA icon (`#1b2a52`/`#5a3a68`/`#e2703a`) for visual continuity across the app. Includes a large Mount Fuji silhouette behind the skyline, offset right of center — genuinely visible from Tokyo on clear days, reuses the same mountain shape as the app icon. Went through several rounds of user feedback: initial size was too small/hidden behind the login card → made much bigger; initial snowcap covered almost the whole peak → shrunk to a proportional cap near the top with the rock body clearly visible below; peak flattened (less pointy) per request; finally offset right so it's not fully hidden behind the centered login card. Heading font is `Yuji_Syuku` (`next/font/google`, a Japanese calligraphy-style face with Latin support) applied only to the "SuiTracker" wordmark, not body text. Animation (window twinkle + drifting mist) was tuned significantly louder after initial pass read as "no animation" — twinkle now 0.15→1.0 opacity over 2.2s with a glow blur filter, mist drift ±14% over 22s, still disabled under `prefers-reduced-motion`. Windows are a full per-building grid (not one dot per building — first pass looked sparse/wrong per user feedback), positions/lit-state computed deterministically (no `Math.random()`, which would cause SSR/client hydration mismatches). The login page has a fixed "paper card on night sky" look regardless of system light/dark mode.
- **Main app theming:** the "leave Map/Agenda unstyled for now" hold was lifted — user asked to theme the main pages too, restrained ("nice but don't overdo it"). Body font switched from Geist Sans to `Plus_Jakarta_Sans` (`app/layout.tsx`) — also fixed a real bug in the process: `globals.css`'s `body` rule was hardcoded to `font-family: Arial, Helvetica, sans-serif`, so the previously-loaded Geist font was never actually applied anywhere; now correctly wired via `var(--font-sans)`. Page background/foreground tokens (`--background`/`--foreground` in `globals.css`) shifted from neutral grey/white to warm-tinted (light `#faf8f5`) / indigo-tinted (dark `#0d1229`, matching the login sky gradient's darkest stop) — extends the same palette established by the app icon/login page rather than a separate look. Primary action buttons (Add day/stop/task, Install) now use a consistent blue accent instead of plain black/white. `NavTabs.tsx` active tab gets a colored underline. `app/trip/layout.tsx` header now shows a small "SuiTracker" wordmark in `Yuji_Syuku` for continuity with the login page. Deliberately did **not** touch card/border colors, category pills, or day colors (already-established data-encoding colors) — kept the change to background, primary font, and CTA/nav accents only, per the "don't overdo it" instruction.
- **App renamed to "SuiTracker"** (was "Japan Trip" in all UI copy — page title, login heading, PWA manifest name/short_name). Cosmetic only, no functional change.
- **Weather badges on stops.** Every stop with a `day_id` (and a joined `day_date`) gets a small weather badge on its `StopCard`, fetched from **Open-Meteo** (`hooks/useWeather.ts`) — free, no API key, no billing, and no relation to the Google Maps API/billing at all (uses coordinates already stored on `stops`, doesn't call Maps in any way). One batched request covers every unique coordinate across the whole trip (not one call per stop) — see `useWeather`'s `relevantStops`/`uniqueCoords`/fingerprinted `queryKey` for how it dedupes and avoids unnecessary refetches. Forecasts only exist ~16 days out, so most stops show nothing until close to their date — expected, not a bug. Clicking the badge opens `components/agenda/WeatherPopover.tsx`: current conditions plus an hour-by-hour strip for that stop's day. The strip is a horizontally swipeable row (`app/globals.css`'s `.scroll-x-subtle` utility — no visible scrollbar on touch devices, a slim styled one on mouse/trackpad devices via `hover: hover`/`pointer: fine`), not a vertical list — a vertical `overflow-y-auto` list inside the modal caused a real layout bug (the modal, being a flex child of the centered backdrop, grew past `max-w-sm` instead of scrolling, because flex items don't shrink below content width without `min-w-0` — fixed by adding `min-w-0` to the modal card). Hours before the current moment are hidden for *today only* — computed via `Intl.DateTimeFormat` explicitly anchored to `Asia/Tokyo`, not the viewer's own device timezone, since the itinerary describes what's happening in Japan regardless of where the app is being checked from. New types: `WeatherPoint`, `HourlyForecastPoint`, `StopWeather` (`lib/types.ts`); icon/label lookup table `WEATHER_CODE_ICONS` (`lib/constants.ts`) for Open-Meteo's WMO weather codes.
- **Cache-shape versioning:** the offline `localStorage` persister (`QueryProvider.tsx`) now sets a `buster` string on `persistOptions`. Bump it any time a persisted query's return shape changes (like `useWeather`'s did, twice, mid-build) — otherwise a returning user's stale cached shape can crash a component expecting the new shape (hit this directly: `Cannot read properties of undefined (reading 'weatherCode')`).
- **Cross-page staleness bug, fixed:** deleting days on the Agenda page didn't reliably show up on the Map page's legend/pins on the next visit within the same session. Root cause: `useTripData()` relied on `staleTime`/`invalidateQueries` timing to decide whether to refetch on mount, which isn't reliable across route changes. Fixed by setting `refetchOnMount: "always"` — every mount now force-refetches from Supabase (still paints instantly from the persisted cache first, then reconciles), guaranteeing same-session edits are never stale on the next page you visit. This is different from the deliberate "manual refresh is fine for syncing a partner's changes" tradeoff — that's about a *different device*; seeing your own just-made edit on the very next page in the same browser should never require a manual refresh.

## Adding a temporary/test user

Public signup is disabled by design, so a test account has to be created the same admin way the original 2 accounts were:

1. Supabase Dashboard → **Authentication → Users → Add User** — any email/password works; dashboard-created users are auto-confirmed (no verification email sent). This doesn't weaken the "no public signup" guarantee since it's an admin action, not an open signup form.
2. That's sufficient to test login/install/PWA behavior. The new account will **not** see any trip data by default (RLS blocks it) — it'll just show "No trip found for your account yet."
3. To let it see/edit the real itinerary too: get the trip's `id` from Supabase (`select id, name from public.trips;` in the SQL Editor, or Table Editor → `trips`), get the new user's UUID from Authentication → Users, then in the SQL Editor:
   ```sql
   insert into trip_members (trip_id, user_id)
   values ('<trip-id>', '<new-user-uuid>');
   ```
4. Delete the user from Authentication → Users when done testing, to get back to exactly 2 real accounts.
